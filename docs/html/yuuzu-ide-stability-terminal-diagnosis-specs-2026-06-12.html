Diagnosis Report + Fix Specs

Yuuzu-IDE 卡頓與 Terminal 問題診斷

逐層程式碼追查（無猜測、無 subagent）· 2026-06-12 · 範圍：src-tauri / src / tauri 2.11.2 / wry 0.55.1 / portable-pty 0.9.0 原始碼

TL;DR

症狀 1「容易卡著、等一陣子才恢復」：105 個 Tauri command 中 95 個是同步函式，而 macOS 上同步 command 會在 main thread（UI 執行緒）執行到完成 —— git fetch/pull/push 走網路、LSP 請求最長阻塞 10 秒、全 workspace 搜尋掃描，任何一個慢就凍結整個視窗。另外 terminal 輸出超過 120k 字元後，前端每收一個 4KB chunk 就觸發 clear() + 全量 120KB 重寫，輸出量大時 webview 直接飽和。

症狀 2「terminal 沒顏色、吃不到指令」：spawn shell 時既不是 login shell、也沒設定 TERM。你的 Homebrew PATH 設定在 ~/.zprofile（只有 login shell 會載入）→ brew 安裝的指令全部 command not found；打包後由 Finder 啟動時 TERM 為空 → 程式判定不支援色彩。PTY 還永遠固定 80×24、沒有 resize 通道。

關鍵發現

已證實同步 command 在 macOS main thread 執行（五層證據鏈，每層皆無執行緒切換）
  1. wry 0.55.1 wkwebview/class/wry_web_view_delegate.rs:35-61：IPC 進入點是 WKWebView 的 WKScriptMessageHandler.didReceiveScriptMessage，WebKit 規範保證在 main thread 呼叫（delegate 以 MainThreadMarker 建立），handler 內直接呼叫 ipc_handler(r)，無 dispatch。
  2. tauri 2.11.2 ipc/protocol.rs:75,311：直接呼叫 webview.on_message(...)，無 spawn。
  3. tauri 2.11.2 webview/mod.rs:1742→1909：on_message 同步呼叫 manager.run_invoke_handler(invoke)，無 spawn。
  4. tauri-macros command/wrapper.rs:404-436（body_blocking）：同步 command 展開為 let result = $path(...) —— 在當前執行緒直接跑完才回應。
  5. yuuzu-ide commands.rs：105 個 command，僅 10 個 async fn（SSH/SFTP/DB），其餘 95 個全部同步。

結論：任何同步 command 的執行時間 = 整個視窗 UI 的凍結時間。command 結束，UI 恢復 —— 與「卡著、等待一段時間才會恢復」的描述完全一致。

已證實最常觸發凍結的三類同步操作：LSP（10s）、git 網路、全域搜尋掃描
  • lsp.rs:17 DEFAULT_LSP_REQUEST_TIMEOUT_MS = 10_000；lsp.rs:617-654 以 recv_timeout 同步阻塞等回應。hover / completion / diagnostics / definition 等 14 個 lsp_* command 全部同步 → 編輯器中移動游標、打字就可能觸發，單次最長凍結 10 秒。首次開檔的 initialize（lsp.rs:990-1026）同樣阻塞。
  • git.rs:261-287 fetch / pull / push → run_git → std::process::Command::output()（git.rs:509-532）—— 網路慢時阻塞數秒到數十秒。
  • search.rs:61-79 以 ignore::WalkBuilder 同步走訪整個 workspace 並逐檔搜尋；scan_workspace / docs_index 同類。
已證實Terminal 輸出管線在 120k 字元後進入 O(n²) 重寫風暴

資料流：PTY reader thread 每 ≤4096 bytes emit 一個 IPC event（terminal.rs:201-218，無聚合）→ AppShell listener（AppShell.tsx:2545）→ zustand store 字串累加並於 120k 處從頭部截斷（terminal-model.ts:20-25）→ React 重新 render → TerminalTab.tsx:103-111 以 output.startsWith(previous) 判斷增量。

頭部一旦被截斷，startsWith 永遠為 false → 之後每個 chunk 都走 fallback：terminal.clear() + 重寫整段 120KB。高輸出指令（build、cat 大檔）下每秒數十至數百個 event × 120KB ANSI 重新解析 + React 全量 re-render → webview 主執行緒飽和、UI 卡死，直到輸出結束才消化完 backlog。

已證實shell 環境缺失：非 login shell、無 TERM —— 與本機 ~/.zprofile 交叉驗證

terminal.rs:161-162 僅 CommandBuilder::new(&info.shell) + cwd，無 -l 參數、無任何 env 設定。portable-pty 的 get_base_env()（cmdbuilder.rs:74-101）只繼承父行程環境再補 SHELL，不會設定 TERM。

本機驗證：~/.zprofile:7 為 eval "$(/opt/homebrew/bin/brew shellenv)" —— macOS 的 zsh 只有 login shell 才載入 ~/.zprofile（Terminal.app / iTerm2 / VS Code 預設都開 login shell，所以平常正常）。Yuuzu-IDE 開的是 non-login zsh → /opt/homebrew/bin 不進 PATH → brew 安裝的指令全部 command not found（「吃不到指令」）。

顏色：打包後由 Finder/launchd 啟動時環境無 TERM → ls / git pager / ncurses 程式判定終端無色彩能力 → 全部單色。dev 模式（bun tauri dev 從終端啟動）會「碰巧」繼承 TERM，症狀較輕 —— 但 PATH 缺失依然存在，且依賴 brew 工具的 prompt 主題 / alias（如 starship、eza）會因 not found 而失效，同樣表現為「沒有顏色」。

已證實PTY 永遠 80×24：spawn 硬編碼、後端無 resize command

AppShell.tsx:3516-3517 spawn 固定 rows: 24, cols: 80；lib.rs:119-123 註冊的 terminal command 只有 probe / list / spawn / write / close，沒有 resize。前端 FitAddon 只調整 xterm 視圖，PTY 與 shell 認知的尺寸永遠是 80×24 → 輸入超過 80 字元時 zsh 行編輯回顯錯亂（觀感像「指令吃不到」）、vim / less / fzf 等 TUI 繪製錯位。

問題 1：「很容易卡著，需要等待一段時間才會恢復」

兩條獨立路徑都會造成此症狀，分別位於 Rust 後端與 React 前端。實際卡頓可能是兩者疊加。

根因 A：同步 command 佔用 macOS main thread（IPC 五層證據鏈）

為什麼是「等待一段時間才會恢復」

  • main thread 被 sync command 佔用期間，視窗完全無法繪製與回應輸入（包含滑鼠、鍵盤、其他面板）。
  • command 一返回（成功、失敗或 LSP 10 秒逾時），main thread 解凍，排隊的 UI 事件瞬間消化 → 「恢復」。
  • 「很容易」卡：LSP hover / completion 在編輯器中是高頻自動觸發，等同每次互動都在賭 language server 的回應速度。

根因 B：Terminal 輸出 O(n²) 重寫風暴（前端 webview 卡死）

// terminal-model.ts:20-25 —— 超過 120k 即從「頭部」截斷
const MAX_OUTPUT_CHARS = 120_000;
function appendBoundedOutput(previous, chunk) {
  const output = `${previous}${chunk}`;
  return output.slice(Math.max(0, output.length - MAX_OUTPUT_CHARS));
}

// TerminalTab.tsx:103-111 —— 增量判斷依賴前綴不變
if (output.startsWith(previous)) {
  terminal.write(output.slice(previous.length));   // 正常增量
} else {
  terminal.clear();                                 // 截斷後永遠走這裡：
  terminal.write(output);                           // 每個 chunk 全量重寫 120KB
}
  • 後端 reader thread 每 ≤4096 bytes 就 emit 一個 IPC event（terminal.rs:201-218），無聚合節流。
  • 每個 event：zustand store 更新（字串拷貝 O(n)）→ React re-render → startsWith 比較 O(n)。
  • 累積輸出超過 120k 後，每個 chunk 額外觸發 clear + 120KB ANSI 全量解析。跑 build、cat 大檔等指令時 webview 主執行緒飽和 → 整個 UI（不只 terminal）卡死，輸出停止後才恢復。
如何區分使用者遇到的是 A 還是 B：卡頓發生在 terminal 跑高輸出指令時 → B；發生在編輯器 hover / 開檔 / git 操作 / 全域搜尋時 → A。兩者皆為 P0/P1 等級，需都修。

問題 2：「terminals 沒有顏色且吃不到一些指令」

「沒顏色」與「吃不到指令」是同一個根因（C）的兩個面向，再由根因 D 放大觀感問題。

根因 C：spawn 的 zsh 既非 login shell、環境變數又殘缺

// terminal.rs:161-162 —— spawn 時只設了 cwd
let mut command = CommandBuilder::new(&info.shell);  // 直接執行 /bin/zsh，無 -l
command.cwd(&cwd);
// 沒有 command.env("TERM", ...)、沒有 COLORTERM、沒有 login shell 旗標

portable-pty 的 CommandBuilder::new（cmdbuilder.rs:215-223）呼叫 get_base_env()：只複製父行程（IDE 本身）的環境變數、補一個 SHELL，僅此而已。

「吃不到指令」的完整鏈路（已用本機設定檔驗證）

  1. macOS 的 zsh 啟動順序：login shell 才載入 /etc/zprofile（內含 path_helper）與 ~/.zprofile；non-login interactive 只載入 ~/.zshrc。
  2. 本機 ~/.zprofile:7 = eval "$(/opt/homebrew/bin/brew shellenv)" —— Homebrew 官方安裝程序就是寫到這裡。
  3. Yuuzu-IDE spawn 的是 non-login zsh → ~/.zprofile 不執行 → /opt/homebrew/bin 與 /opt/homebrew/sbin 不在 PATH。
  4. 結果：所有 brew 安裝的 CLI（node、rg、gh、starship、eza…）→ command not found。~/.zshrc 裡的 PATH 追加（bun、pyenv 等）仍有效，所以「部分指令可用、部分吃不到」—— 與回報症狀吻合。
  5. 打包成 .app 後更糟：launchd 給的基底 PATH 只有 /usr/bin:/bin:/usr/sbin:/sbin。

「沒有顏色」的兩條疊加路徑

  • TERM 未設定：portable-pty 不補 TERM、spawn 程式碼也沒設。由 Finder 啟動的 .app 環境裡根本沒有 TERM → ls、git pager、ncurses 程式查 termcap 失敗 → 判定不支援色彩 → 單色輸出。（dev 模式從終端啟動會繼承 TERM，故 dev 與打包行為不一致。）
  • PATH 連鎖效應：若 prompt 主題 / 著色工具是 brew 安裝（starship、zsh-syntax-highlighting 等），non-login shell 下 not found → prompt 退回 zsh 無色預設。
  • 前端排除：xterm.js 主題已設定 16 色/256 色支援正常（TerminalTab.tsx:51-56 只自訂背景/前景，不影響 ANSI 色），顏色問題不在前端 —— 是 PTY 內程式根本沒輸出 ANSI 色碼。

根因 D：PTY 固定 80×24、無 resize 通道（放大「吃不到指令」的觀感）

  • AppShell.tsx:3516-3517：spawn 永遠傳 rows: 24, cols: 80。
  • lib.rs:119-123：後端只註冊 probe / list / spawn / write / close —— 沒有 resize command；terminal.rs 的 TerminalProcess 也沒保存 master PTY handle，目前結構根本無法 resize。
  • FitAddon 把 xterm 視圖撐到實際大小（如 220×50），但 shell/TUI 認知仍是 80×24 → 輸入超過 80 字元時 ZLE 回顯折行錯亂、補全選單畫錯位置 —— 操作起來「像是指令吃不進去」。

根因 E（次要）：convertEol: true

TerminalTab.tsx:48 對真 PTY 輸出設 convertEol: true 是錯的 —— PTY 行尾已是 CRLF；此選項是給「非 PTY 來源」用的。搭配 clear + 全量 replay 還會破壞 alternate screen（vim/less）狀態。

修正規格（僅規格，未實作）

建議實作順序：B → C → D → A → E（B/C 投入小、立即解除最痛症狀；A 影響面最大、分批遷移）。

P0Spec B：Terminal 輸出管線重構 —— chunk 直寫 xterm，脫離 React render 路徑

  • 變更點 1（TerminalTab.tsx）：terminal-output event 的 chunk 直接 terminal.write(chunk)（透過 ref / 訂閱，不經 React state→prop）。掛載時以 store 內 scrollback replay 一次，之後僅吃增量。
  • 變更點 2（terminal-model.ts）：store 的 outputBySessionId 僅作為「重新掛載時的 replay buffer」，更新不再驅動 TerminalTab render（移出 React 訂閱範圍，或以 subscribe + ref 實作）。
  • 變更點 3：刪除 startsWith / clear() 全量重寫邏輯（TerminalTab.tsx:103-111）。
  • 變更點 4（可選優化）（terminal.rs:201-218）：reader thread 以 8–16ms 視窗聚合多個 4KB chunk 再 emit，降低 IPC event 頻率。
驗收：在 terminal 執行 seq 1 2000000 或 cat 一個 >10MB 檔案期間，UI 其他面板可正常點擊、terminal 可捲動；輸出結束後 prompt 立即可輸入。輸出總量超過 120k 字元後行為不變（無 clear 閃爍、無卡頓）。

P0Spec C：shell 以 login shell 啟動並補齊終端環境變數

  • 變更點（terminal.rs:161-162 spawn_session，SSH terminal 同步比照）： 
    let mut command = CommandBuilder::new(&info.shell);
#[cfg(not(windows))]
command.arg("-l");                          // zsh/bash/fish 皆支援 login 旗標
command.env("TERM", "xterm-256color");      // xterm.js 完整支援
command.env("COLORTERM", "truecolor");
command.env("TERM_PROGRAM", "yuuzu-ide");
// LANG 若父環境缺失則補 en_US.UTF-8（避免打包後 UTF-8 處理異常）
command.cwd(&cwd);
  • Windows 路徑（COMSPEC/powershell）不加 -l，維持現狀。
  • 注意 pty.rs:15 spawn_shell_probe 與名稱顯示（registry 內 "zsh {n}"）不受影響。
驗收：IDE terminal 內 echo $PATH 包含 /opt/homebrew/bin；brew --version 正常執行；echo $TERM 輸出 xterm-256color；ls -G 與 git log 有顏色。打包後的 .app（由 Finder 啟動）重測以上四項皆過 —— dev 模式通過不算數。

P1Spec D：新增 PTY resize 通道

  • 後端（terminal.rs）：TerminalProcess 增存 master: Box<dyn MasterPty + Send>（目前 spawn 後 master 未保存，僅取出 reader/writer）。新增 command resize_terminal_session(session_id, rows, cols) → master.resize(terminal_size(rows, cols))，並於 lib.rs 註冊。
  • 前端：spawn 時改用 fitAddon.proposeDimensions() 的實際值取代硬編碼 80×24（AppShell.tsx:3516）；TerminalTab 的 ResizeObserver 在 fit() 後 debounce（~100ms）呼叫 resize command。
驗收：拖拉視窗大小後在 terminal 執行 tput cols 數值隨之改變；vim 滿版顯示；輸入超過 80 字元的指令時回顯不錯亂。

P1Spec A：耗時同步 command 遷移至 async（分批）

  • 模式：簽名改 pub async fn，同步邏輯包進 tauri::async_runtime::spawn_blocking(move || ...)，State 內容物以 Arc clone 進 closure（現有 State 多為 Arc<Mutex<_>>，成本低）。Tauri async command 在 tokio runtime 執行，不佔 main thread。
  • 第一批（網路/最長阻塞）：git_fetch、git_pull、git_push、全部 14 個 lsp_*。
  • 第二批（IO 重）：search_workspace、scan_workspace、scan_directory、docs_index、docs_search、git_status、git_commit_graph、git_diff_file、run_workspace_task、browser_capture_preview。
  • 第三批：其餘 git 寫操作、read_text_file / write_text_file（大檔）、其餘掃描類。純記憶體讀取類（list_* / settings）可保留同步。
  • 搭配調整：LSP 互動型請求（hover/completion）timeout 由 10s 降至 1–3s（lsp.rs:17 拆分常數）；LSP state 為全域 Mutex，async 化後留意同一 server 的請求仍會串行 —— 可接受，但勿在持鎖時跨 await。
驗收：以 kill -STOP <lsp-server-pid> 模擬 LSP 無回應，在編輯器中 hover / 打字 UI 不凍結；對慢速 remote 執行 git fetch 期間可正常切換面板與捲動。

P2Spec E：移除 convertEol: true（併入 Spec B 實作）

  • TerminalTab.tsx:47-57 的 Terminal 選項移除 convertEol: true —— PTY 輸出已含 CRLF；保留會在邊界情況造成多餘換行。
驗收：多行輸出、vim / less 進出 alternate screen、Ctrl-C 中斷後畫面皆正常。

誠實標註：推論強度

  • 已證實程式碼層事實全部直接讀自原始碼（含 tauri / wry / portable-pty 本機 vendored 原始碼）與本機 ~/.zprofile。
  • 推論「實際卡頓由 A 或 B 哪條觸發」未經執行期重現（本次未跑 app）。建議實作前先用上述驗收手法重現一次，確認權重。卡頓若也發生在「無 LSP、無 terminal 輸出」的場景，優先懷疑 A 的 git_status / 掃描類高頻呼叫。
  • 推論「沒顏色」在 dev 模式下的成因偏向 PATH 連鎖（prompt 工具 not found）；打包後則必然含 TERM 缺失。Spec C 同時覆蓋兩者。
