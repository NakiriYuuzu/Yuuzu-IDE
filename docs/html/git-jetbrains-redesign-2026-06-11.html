Git Deep Dive — JetBrains 等級 Git 功能設計

Yuuzu-IDE · 2026-06-11 · 設計語言：docs/ui-design/ide.css（dark + yuzu-green）· 對應差距審查 git-feature-jetbrains-comparison-2026-06-11

TL;DR

本提案把現有 Node 4 git 功能升級到 JetBrains 等級，共 6 個 surface：① Git Log（真實 DAG 拓撲圖 + 過濾 + commit 詳情 + 右鍵操作 + 單次 commit 內容匯出）、② Partial Staging（hunk/行級暫存）、③ Diff Viewer（side-by-side + word-level 高亮）、④ Merge Conflict 三向解決、⑤ Branch Popup 與 Stash 管理、⑥ Blame 與 File History。所有 mockup 直接以 ide.css 的 tokens 與元件 class（.row/.dbgrid/.menu/.badge2/.btn/.segmented）組成，零新視覺語彙；後端維持「Rust 擁有邏輯 + bounded payload + 打字確認 guardrails」的既有架構紀律。文末附 17 個新 Tauri 命令規格、lane 分配演算法與 roadmap 排程建議。

① Git Log / DAG ② Partial Staging ③ Diff Viewer ④ Conflict 解決 ⑤ Branch / Stash ⑥ Blame / History ⑦ 技術規格

①+ Export Commit — 下載單次 commit 的內容（含資料夾結構）

從右鍵選單或詳情面板觸發。兩種 scope：Changed files only（僅該次 commit 變更的檔案，依原始資料夾結構寫出——「僅當次內容」的預設語意）與 Full snapshot（該 commit 當下的完整專案樹）。輸出可選直接寫成資料夾或打包 Zip。

1Changed files only（預設）：Rust 以 git diff-tree --no-commit-id --name-status -r <hash> 取得變更清單，逐檔 git show <hash>:<path> 取出該 commit 版本的內容寫到目的地，巢狀資料夾自動建立；狀態 D（刪除）的檔案自動略過並在 toast 註記。
2Full snapshot：整棵樹直接走 git archive --format=zip <hash>（git 原生、零自製打包邏輯）；選 Folder 時解開到目的地。
3目的地與安全：Browse 走 @tauri-apps/plugin-dialog（已在依賴中）；寫入路徑必須位於使用者選定目錄內（沿用 git.rs 的路徑消毒模式）；目的地已有同名檔案時需逐次 Overwrite 確認，絕不靜默覆蓋。
4Bound：changed-files 模式上限 2,000 檔 / 200 MB，超限時引導改用 snapshot zip；binary 檔原樣位元組寫出（git show 本就回 raw bytes）。匯出在 Rust 非同步執行，進度回報到 status bar，完成丟 toast + Reveal in Finder。
5入口：① 右鍵選單「Export Commit…」、② 詳情面板 Export… 按鈕、③ command palette「Git: Export Commit…」（作用於 Log 目前選中的 commit）。

資料模型升級

  • git_diff_file 從 raw string 升級為結構化 GitDiffHunks { hunks: [{ header, old_start, new_start, lines: [{ kind, old_no, new_no, text }] }] }，side-by-side 對齊與行號由前端零成本渲染。
  • Word-level：Rust 端對 del/add 配對行跑 LCS（字元 token 化、上限 400 字元/行），輸出 ranges;超限自動退回整行高亮。

Bounded 策略（沿用架構紀律）

  • 每檔 hunk 上限 200、總行 8,000，超限回 truncated: true + 提示列。
  • 編輯器 gutter 變更標記（add/mod 色條，ide.css 已有 .gutter .ln.add/.mod 樣式）由同一份 hunk 資料驅動，hover 顯示 hunk 浮層（revert/stage/copy）。

④ Merge Conflict — 三向解決

偵測到 conflict（現有 has_conflicts 已會擋 commit）時，衝突檔在編輯區開啟三欄解決器：Ours（HEAD）、Result（可編輯）、Theirs（MERGE_HEAD）。逐一 Accept 後 Mark Resolved 自動 git add。

Blame 後端

  • git_blame_file：git blame --porcelain 解析為 [{ hash, author, when, line_start, line_count }] 區段（同 commit 連續行合併），單檔上限 20,000 行，超限回 truncated。
  • annotate 欄是 Monaco 旁的獨立 DOM 欄（同步捲動），不進 Monaco model — 維持懶載與 state 紀律。

File History 後端

  • git_file_history：git log --follow --name-status -- path（分頁同 ①），列出該檔每次變更;點列開「該 commit 中此檔」的 diff（git_commit_file_diff）。
  • --follow 能跨 rename 追蹤，呼應 status 已有的 original_path 欄位。

⑦ 技術規格 — 新增命令、lane 演算法、roadmap 排程

Lane 分配演算法（取代 ASCII 解析）

  • 輸入：topo-order 的 { hash, parents[] } 串流（新→舊）。
  • 維護 active: Vec<Option<Hash>>（各 lane 正在等待的 parent）。每筆 commit：在 active 中找自己 → 該位即本列 lane（找不到則佔第一個空位）。
  • 第一 parent 接替原 lane；其餘 parent 各佔新 lane（合併邊）；多個 lane 同時等同一 parent 時收斂（分支點），釋出右側 lane。
  • 輸出 per-row：{ lane, dot_color, edges: [{from_lane, to_lane, kind}] }，前端每列一個 36px SVG，按 edges 畫貫穿線/曲線（本設計稿 mockup 即此渲染）。
  • lane 上限 12（超出折疊進最右 lane 並淡化）——對個人 repo 足夠且維持 bounded。在 Rust 端計算並隨分頁回傳，前端零圖演算法。

Roadmap 排程建議

  • 建議立為獨立節點「Node 4.5: Git Deep Dive」排在 Node 10（Remote）之後、Node 11（Debugging）之前——依賴面最小，且 Node 7 agent 工作流可立即受益（cherry-pick/blame 作為 agent context）。
  • 任務切分（各自可 TDD）：T1 結構化 diff/hunks ＋ stage/unstage hunk → T2 log 拓撲 ＋ lane 演算法 ＋ 分頁 → T3 commit 詳情 ＋ log 操作（cherry-pick/revert/reset/export）→ T4 branch popup ＋ stash 全套 → T5 conflict 三向 → T6 blame ＋ file history → T7 驗證與量測。
  • Lane 演算法是純函式 — RED 測試直接用合成 DAG（線性、單分支、八爪 merge、criss-cross）斷言 lane/edges。
  • 量測目標：1,000-commit repo 的 log 首頁 < 150 ms；10,000 行檔 blame < 400 ms;全部 payload 有 truncated 旗標。

不做（維持 JetBrains 對齊但尊重 roadmap non-goals）：PR/code review 整合（roadmap 未列）、submodule UI、GPG 簽章 UI、reflog 瀏覽器——若日後需要，全部可沿用 ① 的 log 表格與 ⑤ 的 menu 模式。
