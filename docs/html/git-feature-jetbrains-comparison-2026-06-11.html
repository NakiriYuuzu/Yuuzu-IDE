Feature Audit

Yuuzu-IDE Git 功能細節 vs JetBrains IDE

逐檔人工審閱（無 subagents）· 2026-06-11 · 範圍：src-tauri/src/git.rs、src/features/git/**、AppShell git 串接

TL;DR — 形似、神不似

Yuuzu-IDE 的 git 功能覆蓋了「日常 edit → stage → commit → push 迴圈」，概念佈局與 JetBrains 類似（Source Control 面板、commit graph 視圖、diff 視圖、branch 操作、destructive guardrails），但深度差距很大。Commit 面板比較接近 VS Code 的原生 staging 模型而非 JetBrains 的 changelist 模型；所謂「Commit Graph」實際上是 git log --graph 文字解析出來的靜態表格 + 裝飾性 lane 圓點（上限 120 筆、lane 壓到 3 軌、無 parent 拓撲連線、無法點擊 commit 做任何操作），與 JetBrains 真正的互動式 DAG 視覺化是不同等級的東西。Diff 只有唯讀 unified 文字著色，無 side-by-side、無 hunk staging。這個範圍與 roadmap Node 4 自訂的 acceptance／non-goals 一致——是刻意的階段性精簡，不是走歪。

後端：src-tauri/src/git.rs（929 行）

實作方式：薄封裝 git CLI，不是 libgit2/gitoxide

每個操作都是 Command::new("git").arg("-C").arg(repo_root)... 子行程呼叫，解析 stdout。與 JetBrains（內建 git4idea 引擎 + 深度快取）和 VS Code（git CLI + 大量快取層）相比，這是最薄的封裝，但搭配了嚴格的輸入消毒與 payload 上限：

  • 路徑安全：normalize_repo_relative_paths() 拒絕絕對路徑與 ../ 跳脫（git.rs:403）；所有檔案操作設 GIT_LITERAL_PATHSPECS=1，連 *.txt 這種字面檔名都有測試覆蓋。
  • Payload 上限：diff 240 KB 截斷 + 標記（git.rs:8）、graph 120 筆（git.rs:290）——對應「不把大資料灌進 React state」的架構約束。
  • 破壞性操作確認：require_confirmation() 要求精確文字 —— DISCARD、CHECKOUT {branch}、RESET HARD、REBASE {target}（git.rs:500）。
16 個操作完整清單
  • 狀態：git_status — porcelain v1 -z -b 解析：branch、upstream、ahead/behind、rename 原路徑、7 種 change kind（含 conflict 偵測表 DD/AU/UD/UA/DU/AA/UU）。
  • Diff：git_diff_file — 單檔、staged/unstaged 二態、--no-ext-diff --no-color、binary 偵測、240 KB 截斷。
  • Staging：git_stage_paths（add）、git_unstage_paths（restore --staged）、git_discard_paths（restore，需確認）。
  • Commit：git_commit — -m 單訊息、amend 旗標、push_after 旗標（Commit & Push 一鍵）。
  • Stash：git_stash — 只有 stash push（含 -u 選項）。沒有 stash list / apply / pop / drop — 存進去就只能用終端機撈回來。
  • Branch：git_list_branches（含 upstream、current、remote 標記）、git_create_branch、git_checkout_branch（需確認文字）。
  • 遠端：git_fetch（--prune）、git_pull（--ff-only，拒絕產生 merge commit）、git_push（無 force、無 upstream 設定）。
  • Graph：git_commit_graph — 見「Graph 實作剖析」分頁。
  • 危險操作：git_reset_hard、git_rebase_onto（非互動式 rebase）。

前端：src/features/git/（約 1,500 行）+ AppShell 串接

GitPanel（405 行）— Source Control 面板
  • 分組顯示 Staged Changes / Changes 兩段（git-model.ts groupGitChanges() 另外追蹤 conflicts 組），每列：A/M/D/U token + 檔名（點擊開 diff）+ 列操作（stage / unstage / discard）。
  • Commit 區：訊息 textarea、Commit、Commit & Push（主按鈕，帶 branch + ahead/behind 標籤）、Amend、Stash。
  • 動作 gating 是純函式：canCommit()（須有訊息 + 有 staged + 無 conflict）、canAmend()、canStash() — 有衝突時整個 commit 路徑被擋下。
  • Branch 區：下拉選單 checkout（觸發確認對話框）、新 branch 輸入框、Fetch / Pull / Push 按鈕。
GitDiffView（107 行）— 唯讀 unified diff
  • 逐行著色：+ 綠、- 紅、@@ hunk 標頭、其餘 context — 純文字 <pre> 渲染。
  • Staged/Unstaged badge、binary 檔提示、截斷提示。
  • 沒有：side-by-side、行號欄、word-level 高亮、hunk 折疊、從 diff 做 partial staging、編輯能力。Diff 結果快取在 diffByKey（per path+staged）。
工作台整合 — 這部分比想像中完整
  • Explorer 與分頁裝飾：decorationMapFromStatus() 把 status 轉成 path → A/M/D/U map，檔案樹列與編輯器分頁都會顯示 git token（AppShell.tsx:5577）。
  • Activity rail badge：changeBadgeCount() 在 Git 圖示上顯示變更數。
  • Status bar：branch 名 + ahead/behind（statusBranchLabel()）。
  • 自動刷新：file watcher 事件觸發 git refresh（但忽略 .git/ 內部變更避免迴圈，git-model.ts:303）；task 跑完（exitCode 非 null）也觸發 refresh — 這對應 JetBrains 的自動 VFS refresh 行為。
  • 確認對話框：modal 要求打字輸入精確確認文字（AppShell.tsx:6126）— 比 JetBrains 的按鈕式確認更嚴格的防呆設計。
  • Command palette：git-reset-hard 與 git-rebase-main 兩個命令 — 注意 rebase 目標硬編碼為 main（AppShell.tsx:5277），無法 rebase 到任意分支。

「Commit Graph」實作剖析 — 為什麼說它不是 JetBrains 那種 graph

渲染對比示意

資料來源：git log --graph 的 ASCII 前綴解析

後端執行（git.rs:288–304）：

git log --graph --decorate=short --date=relative --pretty=format:%H%x00%h%x00%s%x00%an%x00%ar%x00%D -n 120

  • lane 的真相：graph_lane()（git.rs:606）取 ASCII 圖形前綴（* | \ 之類）中 * 字元的位置當 lane 編號。它沒有讀 parent hashes（pretty format 裡沒有 %P），所以前端不可能畫出 commit 之間的連線。
  • merge 的真相：merge = subject.starts_with("Merge ")（git.rs:586）——字串啟發式。squash merge、自訂訊息的 merge commit 不會被標記；反之叫 "Merge…" 的普通 commit 會被誤標。
  • 前端 clamp：GitGraphView.tsx:122 Math.min(lane, 2) — 視覺上最多 3 條軌道；merge 曲線是固定形狀的裝飾路徑（GitGraphView.tsx:147），不指向真正的 parent 列。
  • refs 解析：%D 逗號分割成 badge；HEAD 高亮用 ref.includes("HEAD")，tag 樣式用 ref.startsWith("v") — 同樣是啟發式（非 tag: 前綴判斷）。

結論：這是「帶 graph 風格裝飾的 commit 列表」。要升級成 JetBrains 等級，後端需要改用 %H %P 拓撲資料 + 前端 lane 分配演算法，這是另一個量級的工程。

Node 4 自訂範圍 vs 實際交付

結論：與 roadmap Node 4 的 acceptance 一致，缺的東西多半是「刻意 non-goal 或排在能力圖後期」
  • Node 4 acceptance：「日常 edit-test-commit-push 迴圈」「diff 與狀態自動更新」「破壞性操作需明確確認」——三項皆達成（node-4-git-results.md 記錄 97 Rust 測試 + 97 Bun 測試 + Playwright UI smoke）。
  • Node 4 non-goals 明列：「Complex merge UI」「Pull request integration」——所以衝突解決與 PR 整合缺席是設計決定。
  • 但 roadmap「Full Capability Map」的 Git 段落列了 commit graph、blame、worktree awareness、merge conflict workflow——意味著 blame、真 graph、衝突解決是承諾過、尚未排程的後續工作（roadmap.md:127–138）。
  • 觀察到的兩個小瑕疵：(1) rebase 目標硬編碼 main，repo 用其他主分支名（master/trunk）時此命令無用；(2) stash 只進不出，使用者得開終端機撈回。
