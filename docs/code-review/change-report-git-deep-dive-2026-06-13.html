Why — 關鍵設計決策

計畫與現狀的衝突：command wrappers 全部改 async
計畫寫於 2026-06-11，範例是同步 pub fn；但 2026-06-12 的 Spec A（89983ba）已把所有重 IO command 遷移為 async fn + spawn_blocking。計畫同時要求「follow the exact wrapper style of the surrounding git_* commands」——周圍現在都是 async，所以 24 個新 command 全部照現行模式寫（信任檢查留在 await 前），這正是計畫的本意而非偏離。
實證驅動的計畫修正（每一項都先用 live probe 驗證再寫碼）
  • merge commit 的 diff-tree 輸出為空：plain diff-tree 對 merge 不輸出，detail/export 會拿到空清單。改用 -m --first-parent -M（同時啟用 rename 偵測），對普通 commit 行為不變。
  • %1f vs %x1f：for-each-ref 接受 %1f hex literal，但 stash list --format 走 log pretty 體系，%1f 會輸出字面文字——計畫此處有 bug，實作用 %x1f。
  • word range 期望值數錯：計畫斷言 [[31,53]]，手算（Python 驗證）正確值是 [[30,51]]——共同前綴 30 字元、後綴 2。標記區恰為 "SELECT * FROM users" ↔ USERS_PAGE_SQL, [PAGE_SIZE]，語意正確。
  • export 路徑防護：計畫的 target.starts_with(dest) 不解析 .. 組件（dest/../x 字面上仍以 dest 開頭），改為先過 normalize_repo_relative_paths（拒絕逃逸）再 join，starts_with 留作第二道防線；zip 解壓走 zip crate 的 enclosed_name() zip-slip guard。
  • confirmation short hash 對齊：Rust 端用 rev-parse --short 取得期望文字，與前端 log 的 %h 一致，避免長度不一造成 confirmation 永遠不符。
  • remote branch 偵測：放棄計畫的名稱啟發式（自己標註的已知風險），改跑兩次 for-each-ref（refs/heads / refs/remotes 各一次）明確標記。
Task 8 範圍擴大：AppShell diff 資料流一併遷移
計畫把 AppShell 接線排在 Task 10，但 Task 8 改了 GitDiffView 的 props 後，每個 task 的 bun run build 閘門要求全 repo 可編譯。最小正確解是在 Task 8 同步把 diffByKey 從 GitDiff(raw) 換成 GitDiffHunks、agent context 經 hunksToUnifiedText 重組 raw 文字——agent 餵入內容不變，render 換成結構化資料。
Per-block conflict 解決的落地方式
Rust API 只有 whole-side accept（含 confirmation）與 mark-resolved。逐 block 的 ours/theirs 選擇在前端累積（conflictChoices），Mark Resolved 時由純函式 resolveConflictText 把 working copy 的 marker 區段換成所選行，經既有 write_text_file 寫回再 git_mark_resolved stage。沒選的 block 保留 markers，閘門（剩餘計數）擋住提早 resolve。
「new branch / rename / stash branch」的命名輸入
Tauri webview 沒有 window.prompt；新增一個輕量 name dialog（沿用 git-confirm-dialog 樣式），三個流程共用。Context menu 的 New Branch from Here、popup 的 Rename、stash 的 Branch… 都走它。
