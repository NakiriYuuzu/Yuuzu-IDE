1 What — 改了什麼

AgentZone(終端叢集畫面)header 原本的 ▦ N columns 是唯讀標籤,grid 欄數完全由 canvas 寬度自動決定(azColsForWidth:≥3200→4、≥2000→3、≥760→2、否則 1)。

本次將它改為可互動的分段控制 ▦ │ Auto │ 2 │ 3 │ 4 │:

  • Auto(預設): 維持原本行為,欄數跟著 canvas 寬度自動變;對應「目前自動解析到的欄數」那顆數字會顯示淡綠提示(is-auto)。
  • 點 2 / 3 / 4: 鎖定欄數,grid 立即切到該欄數且不再隨寬度變;被選的那顆顯示實心反白(is-on)。
  • 點 Auto: 清除鎖定,回到自動模式。

關鍵改動

  • v2-model.ts: 新增純函式 resolveAzCols(override, width) — override 優先、否則 fallback 到 azColsForWidth。
  • v2-store.ts: 新增全域 state azColsOverride: number | null 與 action setAzColsOverride。
  • AgentZone.tsx: 接線 store、改用 resolveAzCols、唯讀 span → 分段控制按鈕。
  • yuzu.css: .yz2-az-cols 改成分段容器,新增 .seg / .is-on / .is-auto 樣式。

3 Why — 為什麼這樣設計

  • 為何保留 Auto(而非純手動): 使用者在 4 個 UI 方案中選了「Auto + 2/3/4 分段」。保留自動模式可在小視窗下避免硬塞 4 欄造成擁擠,同時給需要時的手動鎖定能力。
  • 為何抽 resolveAzCols 純函式: roadmap 設計原則「React owns UI rendering only」。把「override 優先、否則 auto」的決策放進 model 層,component 只呼叫,邏輯得以單元測試。
  • 為何 override 放全域 state: azWidth 本來就在 V2State 頂層(AgentZone canvas 全域共用),欄數偏好放一起最自然,不需 per-project 複製。
  • 為何加 is-auto 淡提示: 純分段控制在 Auto 模式下,使用者看不出「自動現在是幾欄」。淡提示用極低成本補足這個資訊缺口,鎖定後即消失,不干擾。

4 Architecture — 改動落在 v2 分層的哪裡

v2 的單向資料流:component → store(state/action)→ model(純邏輯)→ CSS 呈現。本次四個改動點(綠色高亮)落在每一層,彼此職責清楚、無循環依賴。
