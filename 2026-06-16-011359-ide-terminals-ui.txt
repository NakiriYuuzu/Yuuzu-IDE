  確認結論：問題出在「IDE 回報給 PTY 的終端機尺寸不穩定」，不是後端

  先排除一個方向：後端 resize 是正確的。src-tauri/src/terminal.rs:312 的
  resize_session() 確實有呼叫 master.resize()，而且有測試
  resize_session_updates_kernel_pty_size 證明 kernel PTY winsize
  會被正確更新。所以後端沒問題。

  問題在前端 TerminalTab.tsx 回報尺寸的「時機」與「值」。你截圖裡 Claude Code
  的狀態列疊字（attempt 1/10 疊上 tokens · thinking…、Pollinating…———
  被線切過），是典型的 TUI 用「錯誤/晃動的寬度」原地重繪造成的。成因鏈是這樣：

  1. 視覺尺寸「立刻變」，但通知 PTY「慢 100ms 且只在尾端」

  src/features/terminal/TerminalTab.tsx:91

  const resizeObserver = new ResizeObserver(() => {
      fitAddon.fit();                                            // ① xterm 
  顯示寬度「立刻」變
      if (resizeNotifyTimer !== undefined) clearTimeout(resizeNotifyTimer);
      resizeNotifyTimer = setTimeout(notifyPtySize, 100);        // ② 通知後端 
  PTY「100ms 後」才送，且每次 resize 都重置
  });

  這 100ms 空窗裡，xterm 顯示寬度 A、Claude Code 以為自己是寬度 B → 它在 B
  寬度原地重繪，xterm 卻照 A 換行 → 舊字沒被蓋掉就疊上 = 跑版。

  2. 而且尺寸一直在晃，永遠不收斂（這就是「一直」跑版的關鍵）

  fit() 沒有任何節流/變更判斷，每個 ResizeObserver tick 都跑；再加上：
  - AgentZone 視窗有開場動畫 animation: yzIn 0.22s（yuzu.css:2290）+
  grid-auto-rows: minmax(220px,1fr)（yuzu.css:2271）
  - AgentZone 自己第二個 observer 把 canvas 寬度寫回 store（AgentZone.tsx:31 →
  setAzWidth）

  → host 尺寸持續微幅變動 → fit() 每幀跑、但 debounce 的 notify
  一直被往後推/送出晃動值 → PTY 寬度永遠追不上、不穩定 → Claude Code
  每次重繪都用到不一致的寬度 → 持續跑版。

  3. 加重因子：spawn 時硬寫死 rows:24, cols:80（controller.ts:653），且 xterm.js
  在 cols 改變時會 reflow normal-buffer 的 scrollback，跟 Claude Code
  自己的原地重繪互相打架。

  ---
  一句話：yuuzu-ide 的終端機把「改 xterm 顯示」和「通知 PTY
  真實尺寸」拆成兩步、後者還 debounce
  100ms，加上版面持續微抖讓尺寸不收斂，於是像 Claude Code 這種會原地重繪狀態列的
